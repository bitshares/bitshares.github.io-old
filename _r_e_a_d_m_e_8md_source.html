18 BitShares Core is the BitShares blockchain implementation and command-line interface.
19 The web wallet is [BitShares UI](https://github.com/bitshares/bitshares-ui).
20 
21 Visit [BitShares.org](https://bitshares.org/) to learn about BitShares and join the community at [BitSharesTalk.org](https://bitsharestalk.org/).

23 Getting Started
24 ---------------
25 Build instructions and additional documentation are available in the
26 [wiki](https://github.com/bitshares/bitshares-core/wiki).
27 
28 We recommend building on Ubuntu 16.04 LTS (64-bit)
29 
30 **Build Dependencies**:
31 
32  sudo apt-get update
33  sudo apt-get install autoconf cmake make automake libtool git libboost-all-dev libssl-dev g++ libcurl4-openssl-dev
34 
35 **Build Script:**
36 
37  git clone https://github.com/bitshares/bitshares-core.git
38  cd bitshares-core
39  git checkout master # may substitute "master" with current release tag
40  git submodule update --init --recursive
41  cmake -DCMAKE_BUILD_TYPE=RelWithDebInfo .
42  make
43 
44 **Upgrade Script** (prepend to the Build Script above if you built a prior release):
45 
46  git remote set-url origin https://github.com/bitshares/bitshares-core.git
47  git checkout master
48  git remote set-head origin --auto
49  git pull
50  git submodule update --init --recursive # this command may fail
51  git submodule sync --recursive
52  git submodule update --init --recursive
53 
54 **NOTE:** BitShares requires a [Boost](http://www.boost.org/) version in the range [1.57 - 1.65.1]. Versions earlier than
55 1.57 or newer than 1.65.1 are NOT supported. If your system's Boost version is newer, then you will need to manually build
56 an older version of Boost and specify it to CMake using `DBOOST_ROOT`.
57 
58 **NOTE:** BitShares requires a 64-bit operating system to build, and will not build on a 32-bit OS.
59 
60 **NOTE:** BitShares now supports Ubuntu 18.04 LTS
61 
62 **NOTE:** BitShares now supports OpenSSL 1.1.0
63 
64 **After Building**, the `witness_node` can be launched with:
65 
66  ./programs/witness_node/witness_node
67 
68 The node will automatically create a data directory including a config file. It may take several hours to fully synchronize
69 the blockchain. After syncing, you can exit the node using Ctrl+C and setup the command-line wallet by editing
70 `witness_node_data_dir/config.ini` as follows:
71 
72  rpc-endpoint = 127.0.0.1:8090
73 
74 **IMPORTANT:** By default the witness node will start in reduced memory ram mode by using some of the commands detailed in [Memory reduction for nodes](https://github.com/bitshares/bitshares-core/wiki/Memory-reduction-for-nodes).
75 In order to run a full node with all the account history you need to remove `partial-operations` and `max-ops-per-account` from your config file. Please note that currently(2018-07-02) a full node will need more than 100GB of RAM to operate and required memory is growing fast. Consider the following table before running a node:
76 
77 | Default | Full | Minimal | ElasticSearch
78 | --- | --- | --- | ---
79 | 16G RAM | 120G RAM | 4G RAM | 500G SSD HD, 32G RAM
80 
81 After starting the witness node again, in a separate terminal you can run:
82 
83  ./programs/cli_wallet/cli_wallet
84 
85 Set your inital password:
86 
87  >>> set_password <PASSWORD>
88  >>> unlock <PASSWORD>
89 
90 To import your initial balance:
91 
92  >>> import_balance <ACCOUNT NAME> [<WIF_KEY>] true
93 
94 If you send private keys over this connection, `rpc-endpoint` should be bound to localhost for security.
95 
96 Use `help` to see all available wallet commands. Source definition and listing of all commands is available
97 [here](https://github.com/bitshares/bitshares-core/blob/master/libraries/wallet/include/graphene/wallet/wallet.hpp).

109 Using the API
110 -------------
111 
112 We provide several different API's. Each API has its own ID.
113 When running `witness_node`, initially two API's are available:
114 API 0 provides read-only access to the database, while API 1 is
115 used to login and gain access to additional, restricted API's.
116

133 Accessing restricted API's
134 --------------------------
135 
136 You can restrict API's to particular users by specifying an `api-access` file in `config.ini` or by using the `--api-access /full/path/to/api-access.json` startup node command. Here is an example `api-access` file which allows
137 user `bytemaster` with password `supersecret` to access four different API's, while allowing any other user to access the three public API's
138 necessary to use the wallet:
139 
140  {
141  "permission_map" :
142  [
143  [
144  "bytemaster",
145  {
146  "password_hash_b64" : "9e9GF7ooXVb9k4BoSfNIPTelXeGOZ5DrgOYMj94elaY=",
147  "password_salt_b64" : "INDdM6iCi/8=",
148  "allowed_apis" : ["database_api", "network_broadcast_api", "history_api", "network_node_api"]
149  }
150  ],
151  [
152  "*",
153  {
154  "password_hash_b64" : "*",
155  "password_salt_b64" : "*",
156  "allowed_apis" : ["database_api", "network_broadcast_api", "history_api"]
157  }
158  ]
159  ]
160  }
161 
162 Passwords are stored in `base64` as salted `sha256` hashes. A simple Python script, `saltpass.py` is avaliable to obtain hash and salt values from a password.
163 A single asterisk `"*"` may be specified as username or password hash to accept any value.
164 
165 With the above configuration, here is an example of how to call `add_node` from the `network_node` API:
166 
167  {"id":1, "method":"call", "params":[1,"login",["bytemaster", "supersecret"]]}
168  {"id":2, "method":"call", "params":[1,"network_node",[]]}
169  {"id":3, "method":"call", "params":[2,"add_node",["127.0.0.1:9090"]]}
170 
171 Note, the call to `network_node` is necessary to obtain the correct API identifier for the network API. It is not guaranteed that the network API identifier will always be `2`.
172 
173 Since the `network_node` API requires login, it is only accessible over the websocket RPC. Our `doxygen` documentation contains the most up-to-date information
174 about API's for the [witness node](https://bitshares.github.io/doxygen/namespacegraphene_1_1app.html) and the
175 [wallet](https://bitshares.github.io/doxygen/classgraphene_1_1wallet_1_1wallet__api.html).
176 If you want information which is not available from an API, it might be available
177 from the [database](https://bitshares.github.io/doxygen/classgraphene_1_1chain_1_1database.html);
178 it is fairly simple to write API methods to expose database methods.

180 FAQ
181 ---
182 
183 - Is there a way to generate help with parameter names and method descriptions?
184 
185  Yes. Documentation of the code base, including APIs, can be generated using Doxygen. Simply run `doxygen` in this directory.
186 
187  If both Doxygen and perl are available in your build environment, the CLI wallet's `help` and `gethelp`
188  commands will display help generated from the doxygen documentation.
189 
190  If your CLI wallet's `help` command displays descriptions without parameter names like
191  `signed_transaction transfer(string, string, string, string, string, bool)`
192  it means CMake was unable to find Doxygen or perl during configuration. If found, the
193  output should look like this:
194  `signed_transaction transfer(string from, string to, string amount, string asset_symbol, string memo, bool broadcast)`
195 
196 - Is there a way to allow external program to drive `cli_wallet` via websocket, JSONRPC, or HTTP?
197 
198  Yes. External programs may connect to the CLI wallet and make its calls over a websockets API. To do this, run the wallet in
199  server mode, i.e. `cli_wallet -s "127.0.0.1:9999"` and then have the external program connect to it over the specified port
200  (in this example, port 9999).
201 
202 - Is there a way to access methods which require login over HTTP?
203 
204  No. Login is inherently a stateful process (logging in changes what the server will do for certain requests, that's kind
205  of the point of having it). If you need to track state across HTTP RPC calls, you must maintain a session across multiple
206  connections. This is a famous source of security vulnerabilities for HTTP applications. Additionally, HTTP is not really
207  designed for "server push" notifications, and we would have to figure out a way to queue notifications for a polling client.
208 
209  Websockets solves all these problems. If you need to access Graphene's stateful methods, you need to use Websockets.
210 
211 - What is the meaning of `a.b.c` numbers?
212 
213  The first number specifies the *space*. Space 1 is for protocol objects, 2 is for implementation objects.
214  Protocol space objects can appear on the wire, for example in the binary form of transactions.
215  Implementation space objects cannot appear on the wire and solely exist for implementation
216  purposes, such as optimization or internal bookkeeping.
217 
218  The second number specifies the *type*. The type of the object determines what fields it has. For a
219  complete list of type ID's, see `enum object_type` and `enum impl_object_type` in
220  [types.hpp](https://github.com/bitshares/bitshares-2/blob/bitshares/libraries/chain/include/graphene/chain/protocol/types.hpp).
221 
222  The third number specifies the *instance*. The instance of the object is different for each individual
223  object.
224 
225 - The answer to the previous question was really confusing. Can you make it clearer?
226 
227  All account ID's are of the form `1.2.x`. If you were the 9735th account to be registered,
228  your account's ID will be `1.2.9735`. Account `0` is special (it's the "committee account,"
229  which is controlled by the committee members and has a few abilities and restrictions other accounts
230  do not).
231 
232  All asset ID's are of the form `1.3.x`. If you were the 29th asset to be registered,
233  your asset's ID will be `1.3.29`. Asset `0` is special (it's BTS, which is considered the "core asset").
234 
235  The first and second number together identify the kind of thing you're talking about (`1.2` for accounts,
236  `1.3` for assets). The third number identifies the particular thing.
237 
238 - How do I get the `network_add_nodes` command to work? Why is it so complicated?
239 
240  You need to follow the instructions in the "Accessing restricted API's" section to
241  allow a username/password access to the `network_node` API. Then you need
242  to pass the username/password to the `cli_wallet` on the command line or in a config file.
243 
244  It's set up this way so that the default configuration is secure even if the RPC port is
245  publicly accessible. It's fine if your `witness_node` allows the general public to query
246  the database or broadcast transactions (in fact, this is how the hosted web UI works). It's
247  less fine if your `witness_node` allows the general public to control which p2p nodes it's
248  connecting to. Therefore the API to add p2p connections needs to be set up with proper access
249  controls.
